# Slide 9 Specification

## Slide Purpose
This is the most practical and memorable scene in the presentation.

It shows:
1. The different prompting approaches
2. When each should be used
3. A real-world mapping of those approaches
4. A concrete, usable example of the most advanced approach

This slide should feel like:
- a guided discovery
- a deepening understanding (not a reset)
- a seamless transition from concept → reality → execution

---

## Core Idea (locked)

Use the right approach

---

## On-Slide Structure (locked)

Four-column horizontal layout that persists across all stages.

Columns represent:

Quick build → Structured spec → Planned system → Full system design

These are NOT “levels”
They are **context-dependent approaches**

## Expanded Prompt (locked)

### Full System Design Example

Prompt 1:
Write a detailed product spec for an interactive isometric theme park simulation game that runs in the browser. Include park building, navigation, guests, rides, paths, scenery, UI, economy, cleanliness, queueing, and happiness systems. Ask me about key product decisions before finalizing.

---

Prompt 2:
Before building, design a document system that defines how the product is built and maintained.

Create:
- AGENTS.md (rules + workflow for humans and AI)
- architecture.md (system map and source of truth)
- README.md (documentation map)
- accounts.md (external tools and ownership)
- LOCAL_DEV.md (local runbook)
- setup.md (environment setup)

---

For each document:
- define purpose
- define source of truth
- define when it must be updated
- define how it is used

---

Include maintainability rules:
- enforce reuse over recreation
- define boundaries between systems
- keep naming consistent
- update docs when structure changes
- remove outdated systems instead of duplicating

---

Define workflow:
- start from docs before coding
- if a change affects structure, update docs first
- flag conflicts with existing system
- decide whether to adapt request or update system

---

Goal:
Keep the system aligned as it evolves

## Visual Design (locked)

### Layout
- Full-screen
- 4 evenly spaced columns
- consistent cards or containers

---

### Visual Principles
- clean
- minimal
- no clutter
- no memes
- consistent formatting

---

### Hierarchy
- subtle emphasis increases left → right
- final column most visually important

---

### Zoom State
- final column expands to center
- becomes dominant element
- prompt shown in readable card

---

## Animation

### Column Flow
- columns appear
- labels reveal
- prompt text reveals
- prompt → project mapping transition

---

### Mapping Transition
- smooth fade or morph
- feels like “translation”

---

### Zoom
- smooth zoom into final column
- no harsh transitions

---

### Final Reveal
- prompt fades or slides in
- clean, readable
- after returning to the 4-column view, supporting example assets reveal one by one left → right

## Speaker Script (guideline)

“This isn’t about good vs bad…”

“Each of these is actually the right approach—depending on what you’re building.”

---

(point left)

“If you’re building something simple…  
just ask AI to build it.”

---

(next)

“If it’s more structured…  
write a better spec.”

---

(next)

“If it’s a real prototype…  
plan the system.”

---

(pause → last)

“But if you’re building something long-term…”

“This is where everything changes.”

---

(mapping appears)

“Here’s how I’ve actually used each of these.”

Walk across briefly.

---

(point to final)

“Lift & Learn is where everything breaks…  
if you don’t do this properly.”

---

(zoom)

“And this is what it actually looks like.”

---

Explain at a high level:
- define product
- define docs
- define rules
- keep system aligned

---

Final line:

“The goal is not just to build something…”

“It’s to build something that holds up.”

## Do Not

- do not show everything at once
- do not overload text early
- do not break visual consistency
- do not make mapping feel like a new slide
- do not make final prompt feel overwhelming

---

## Success Criteria

The slide is successful if:
- audience understands when to use each approach
- audience sees themselves in one of the columns
- mapping to real examples feels natural
- zoom creates a strong focus moment
- final example feels usable and not intimidating
